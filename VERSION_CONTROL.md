# SuperSolo AI 代码版本更新与回滚制度
---
## 📋 总则
为保障项目代码安全、可追溯、可回滚，规范开发流程，特制定本制度。所有开发人员必须严格遵守。

---
## 🌿 分支管理策略
采用 **Git Flow 简化版** 分支策略，适合小团队快速迭代：

### 1. 长期分支（永久存在）
| 分支名 | 作用 | 权限 | 合并规则 |
|--------|------|------|----------|
| `main` | 生产环境分支，对应线上运行的稳定代码 | 仅管理员可推送/合并 | 只能从 `release` 或 `hotfix` 分支合并 |
| `develop` | 开发环境分支，对应最新的开发进度 | 所有开发人员可推送 | 从 `feature/*` 分支合并 |

### 2. 临时分支（用完即删）
| 分支名格式 | 作用 | 生命周期 | 合并目标 |
|------------|------|----------|----------|
| `feature/xxx` | 新功能开发分支 | 从 `develop` 创建，开发完成后删除 | 合并到 `develop` |
| `release/vx.x.x` | 版本发布分支，用于发布前测试 | 从 `develop` 创建，发布完成后删除 | 合并到 `main` 和 `develop` |
| `hotfix/xxx` | 线上紧急修复分支 | 从 `main` 创建，修复完成后删除 | 合并到 `main` 和 `develop` |

---
## 📝 版本号命名规范
采用 **语义化版本号**：`v主版本号.次版本号.修订号`
- 主版本号：重大架构变更、不兼容的API修改
- 次版本号：新增功能、兼容性功能迭代
- 修订号：Bug修复、小优化

示例：
- `v1.0.0` - 第一个正式上线版本
- `v1.1.0` - 新增了比价功能
- `v1.1.1` - 修复了比价功能的Bug

## 🔄 代码提交规范
### Commit Message 格式
```
<类型>(<范围>): <描述>
```

### 类型说明
| 类型 | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复Bug |
| `docs` | 文档更新 |
| `style` | 代码格式调整（不影响逻辑） |
| `refactor` | 代码重构（无功能变化） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖等非业务代码变更 |

示例：
- `feat(products): 新增爆款卡片收藏功能`
- `fix(auth): 修复登录页验证码过期问题`
- `docs: 更新版本控制规范文档`

## 🚀 版本发布流程
### 1. 常规版本发布
1. 开发完成后，从 `develop` 创建 `release/vx.x.x` 分支
2. 在release分支上进行测试，修复Bug
3. 测试通过后：
   - 合并到 `main` 分支，打版本标签 `vx.x.x`
   - 合并回 `develop` 分支
   - 删除release分支
4. 部署main分支到生产环境

### 2. 紧急修复发布
1. 从 `main` 创建 `hotfix/xxx` 分支
2. 修复Bug并测试
3. 测试通过后：
   - 合并到 `main` 分支，打版本标签 `vx.x.x`（修订号+1）
   - 合并回 `develop` 分支
   - 删除hotfix分支
4. 立即部署main分支到生产环境

## ⏪ 版本回滚制度
### 回滚触发条件
当出现以下情况时，必须执行版本回滚：
1. 生产环境出现严重Bug，影响核心功能使用
2. 新版本导致性能严重下降，用户无法正常使用
3. 出现安全漏洞，需要立即回退

### 回滚流程
1. **紧急止损**：立即通知所有相关人员，暂停新版本发布
2. **评估影响**：确定回滚版本，评估回滚可能带来的影响
3. **执行回滚**：
   - 方式1：使用Git重置到上一个稳定版本标签：`git reset --hard vx.x.x`
   - 方式2：直接部署上一个版本的构建产物
4. **验证回滚**：确认生产环境已恢复正常，功能可用
5. **问题排查**：排查新版本问题原因，修复后重新发布

### 回滚注意事项
- 回滚操作必须由管理员执行，操作前必须备份当前代码
- 回滚后必须记录回滚原因、时间、操作人，写入发布日志
- 问题未彻底修复前，禁止再次发布有问题的版本

---
## 📊 发布日志管理
每次版本发布必须填写发布日志，包含以下内容：
- 版本号
- 发布时间
- 发布人
- 主要更新内容（功能新增/修复/优化）
- 影响范围
- 回滚方案

发布日志存储在 `RELEASE_LOG.md` 文件中，永久留存。

---
## 🔒 权限管理
- 生产环境 `main` 分支：仅管理员有合并权限，禁止直接推送
- 开发环境 `develop` 分支：所有开发人员可推送，但重要合并需要Code Review
- 所有PR合并前必须至少有1人审核通过
